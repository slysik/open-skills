Snowflake Kafka Connector

One-page cheat sheet for real-time JSON ingest into Snowflake.

What it is

A Kafka Connect sink plugin (a JAR) maintained by Snowflake. Drop it into a Kafka Connect worker, give it a config, and it streams Kafka topic messages into Snowflake tables — no app code in between. Two paths exist: classic Snowpipe (file-based, slow) and Snowpipe Streaming (row-based, sub-second). v4 always uses Streaming and ships with schema evolution on by default.

Architecture

Producer apps ─▶ Kafka topic ─▶ Kafka Connect ─▶ Snowflake table (your code) (broker) (runs the JAR) (queryable) v3 classic Snowpipe : buffer ─▶ gz file ─▶ STAGE ─▶ PIPE ─▶ TABLE (~30s lag) v3 / v4 Streaming : Streaming SDK ─▶ CHANNEL ─▶ TABLE (sub-second)

Table layout

Without schematization (classic)

RECORD_METADATA  VARIANT  -- envelope
RECORD_CONTENT   VARIANT  -- raw payload

Query with : accessors: record_content:sensorId::int

With schematization (v4 default)

RECORD_METADATA  VARIANT
SENSORID         NUMBER(38,0)
PSI              NUMBER(38,0)
TS               TIMESTAMP_NTZ
LOCATION         VARIANT  -- nested obj

Top-level keys ➝ typed columns. Nested ➝ VARIANT. Top-level arrays are not supported.

End-to-end in 3 steps

1Provision Snowflake

CREATE DATABASE  kafka_db;
CREATE SCHEMA    kafka_db.kafka_schema;
CREATE WAREHOUSE kafka_wh WITH WAREHOUSE_SIZE=XSMALL AUTO_SUSPEND=60;

CREATE USER kafka_connector_user;
ALTER  USER kafka_connector_user SET RSA_PUBLIC_KEY='<body, no PEM headers>';

CREATE ROLE kafka_connector_role;
GRANT USAGE        ON DATABASE  kafka_db              TO ROLE kafka_connector_role;
GRANT USAGE        ON SCHEMA    kafka_db.kafka_schema TO ROLE kafka_connector_role;
GRANT CREATE TABLE ON SCHEMA    kafka_db.kafka_schema TO ROLE kafka_connector_role;
GRANT USAGE        ON WAREHOUSE kafka_wh              TO ROLE kafka_connector_role;
GRANT ROLE kafka_connector_role TO USER kafka_connector_user;

2Generate key-pair, post connector config

# Encrypted PKCS#8 private key + matching public key
openssl genrsa 2048 \
  | openssl pkcs8 -topk8 -v2 aes256 -inform PEM -out rsa_key.p8 -passout pass:<PP>
openssl rsa -in rsa_key.p8 -pubout -out rsa_key.pub -passin pass:<PP>

# Strip PEM headers + newlines for the connector config
grep -v "BEGIN\|END" rsa_key.p8 | tr -d '\n' > rsa_key.connector

curl -s -X POST -H 'Content-Type: application/json' \
     --data @connector-v4.json  http://localhost:8083/connectors

3Produce, then verify

echo '{"sensorId":1,"psi":451,"ts":"2026-05-03T19:23:51Z","location":{"site":"AUS-1"}}' \
  | docker exec -i kafka kafka-console-producer --broker-list kafka:9092 --topic sensors

snowq "DESCRIBE TABLE kafka_db.kafka_schema.sensors"
snowq "SELECT sensorid, psi, ts, location FROM kafka_db.kafka_schema.sensors LIMIT 5"

Top 5 gotchas (interview bait)

#1 Private key format Strip PEM headers and newlines — the connector wants a single base64 string. grep -v "BEGIN\|END" rsa_key.p8 | tr -d '\n'
#2 v4 grants are direct only GRANT INSERT ON TABLE foo TO ROLE x directly. Role hierarchy is ignored by v4 — this is the #1 silent permissions failure.
#3 Schematization on a pre-existing table Needs OWNERSHIP on the table + ENABLE_SCHEMA_EVOLUTION=TRUE, otherwise the connector silently drops payloads (only RECORD_METADATA is written). Easier path: drop the table and let the connector auto-create it.
#4 errors.tolerance=all + no DLQ = silent data loss Server-side validation failures land in <table>$errors; client-side ones go to the DLQ topic only if you set errors.deadletterqueue.topic.name.
#5 v4 JVM heap should be ~50% The Rust Streaming SDK runs off-heap. Standard Kafka Connect tuning advice (give JVM 80% of RAM) starves it. Set KAFKA_HEAP_OPTS="-Xmx4G -Xms4G" on an 8 GB worker.

Verify queries

-- Row count
SELECT COUNT(*) FROM kafka_db.kafka_schema.sensors;

-- Schema after schematization
DESCRIBE TABLE kafka_db.kafka_schema.sensors;

-- End-to-end Streaming latency
SELECT DATEDIFF('ms',
         TO_TIMESTAMP(record_metadata:CreateTime::bigint/1000),
         TO_TIMESTAMP(record_metadata:SnowflakeConnectorPushTime::bigint/1000)) AS lag_ms
FROM kafka_db.kafka_schema.sensors ORDER BY lag_ms DESC LIMIT 5;

-- Server-side rejected rows
SELECT * FROM kafka_db.kafka_schema.sensors$errors LIMIT 10;

Snowflake objects the connector creates

v3 classic

  • 1 internal stage / topic
    SNOWFLAKE_KAFKA_CONNECTOR_<name>_STAGE_<table>
  • N pipes / topic (one per partition)
    ..._PIPE_<table>_<partition>
  • 1 target table

v4 / Snowpipe Streaming

  • 1 pipe / table
    <TABLE>-STREAMING
  • N channels (one per partition)
    <CONNECTOR>_<id>_<topic>_<partition>
  • No stages, no files
Exactly-once Each Snowpipe Streaming channel persists its offsetTokenUpperBound server-side. Restart the connector and it resumes from the persisted offset — already-ingested records aren't duplicated.
"It's a Kafka Connect sink plugin from Snowflake. v4 streams Kafka partitions directly into Snowflake tables via Snowpipe Streaming channels — sub-second latency, schema evolution by default, exactly-once via persisted channel offsets. Auth is RSA key-pair, the connector role needs direct grants, and the JVM heap should be modest because the Rust SDK runs off-heap."
